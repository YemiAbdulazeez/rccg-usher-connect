# RNUMS — Build Plan (Frontend + Backend, End-to-End)

> RCCG National Ushering Management System — Phase 1 (MVP).
> This document is deliberately blunt. It states what exists, what is broken, and exactly what will be built and in what order.

---

## 0. Locked decisions

| Decision | Choice |
| --- | --- |
| Frontend architecture | **Convert to plain Vite + React + TypeScript SPA.** Strip TanStack Start / Nitro / SSR. Keep TanStack Router (SPA/memory-free browser history mode) + shadcn/ui + Tailwind v4. |
| Auth | **httpOnly cookie JWT.** Access token (short-lived) + refresh token, both httpOnly, `Secure`, `SameSite=None` in prod. CORS with `credentials: true`. |

| Backend stack | Node.js + Express + MySQL (mysql2) + Cloudinary + Paystack + Resend. |

## 1. Brutal assessment of the current frontend

The frontend is a **clickable design prototype, not an application**. It is generated by Lovable as a **TanStack Start** (SSR) app — NOT the "plain Vite React" the brief assumed.

What is fake and must be replaced:

1. **No real auth.** `src/routes/login.tsx` has ~18 hardcoded demo accounts; "login" writes a fake user to `localStorage` (`lib/profileState.ts` → `setCurrentUser`). No password verification, no token, no session. Google/Microsoft buttons just navigate.
2. **No backend / no API layer.** All state lives in `localStorage`:
   - `lib/profileState.ts` → profiles, approval status, audit trail (`rnums.profiles.v1`, `rnums.currentUser.v1`).
   - `app.hostel.tsx` → bookings/limits/event (`rnums.hostel.*`).
   - `app.id-cards.tsx` → applications (`rnums.idcards.applications`).
3. **Fake data model.** `ProfileFields` is a flat bag of strings. Regions/Provinces are literally `"Region 1".."Region 100"` / `"Province 1".."Province 200"`. **No real hierarchy, no FKs** — the exact backbone the PRD needs for approval routing.
4. **Approval routing does not exist.** `app.approvals.tsx` shows *every* pending profile to *any* PHU/RHU. PRD requires auto-routing to the correct PHU (by province) then correct RHU (by region), no manual forwarding.
5. **Payments mocked.** Hostel + ID card "pay" fabricate records. No Paystack.
6. **Uploads are base64 data URLs in localStorage.** No Cloudinary. Breaks at scale.
7. **QR is a third-party image URL** (`api.qrserver.com`). Notifications/email/audit logs are toasts and in-memory arrays — no persistence.
8. **Phase 1 scope violated.** `AppSidebar.tsx` exposes Training, Attendance, Welfare, Store, Finance, Community, etc. Menu-access rules from the PRD are not enforced (a "pending" usher sees the full menu).
9. **Registration wrong vs PRD.** `register.tsx` collects a "Parish" text field, no confirm password, no role, and "Create account" just links to `/onboarding` without creating anything.

### What is genuinely good and will be kept

- The **entire UI/design system**: shadcn components (`src/components/ui/*`), layout, hostel/id-card/approvals/onboarding screens, status badges, pending-approval disclaimer.
- The **status machine** in `profileState.ts` (`incomplete → pending_phu → phu_approved/pending_rhu → approved/rejected/*_correction`). Good model — moves server-side.
- The **route structure** matches PRD modules.

```
┌──────────────────────────┐         HTTPS + cookies (credentials)        ┌──────────────────────────┐
│  Frontend (Vite SPA)     │  ───────────────────────────────────────▶   │  Backend (Express API)   │
│  React 19 + TS           │  ◀───────────────────────────────────────    │  Node + Express          │
│  TanStack Router (SPA)   │        JSON REST /api/v1/*                    │  MySQL (mysql2/pool)     │
│  TanStack Query          │                                              │  JWT (httpOnly cookies)  │
│  shadcn/ui + Tailwind    │                                              │  Cloudinary/Paystack/    │
│  Axios client (creds)    │                                              │  Resend                  │
└──────────────────────────┘                                              └──────────────────────────┘
```

- Frontend and backend are **separate deployables**, different origins.
- All business logic lives in the **Express backend**. The frontend never touches MySQL/secrets.
- Auth via httpOnly cookies → protects tokens from XSS. Requires strict CORS allowlist + CSRF consideration (see §9).

## 4. Database schema (MySQL) — the 15 core tables

All tables: `id` BIGINT UNSIGNED AUTO_INCREMENT PK, `created_at`, `updated_at` TIMESTAMP, soft-delete `deleted_at` NULL where relevant. Money in **kobo** (INT) to avoid float errors. Charset `utf8mb4`.

### 4.1 `roles`
`id, name (unique), slug (unique), rank (int, hierarchy order), self_registerable (bool)`
Seed all roles from the PRD hierarchy. `self_registerable = true` only for: parish head usher, assistant parish head usher, area head usher, assistant area head usher, zonal head usher, assistant zonal head usher, usher. Everything PHU-and-above = false.

> Note: PRD uses "PHU" for **Provincial** Head Usher in the approval flow (routes by province) and separately lists "Parish Head Usher". Approval **stage 1 = Provincial Head Usher**, **stage 2 = Regional Head Usher**. This is wired explicitly in §6.

### 4.2 `regions`
`id, name, code (unique), rhu_user_id (FK users NULL)` — the RHU responsible for this region.

### 4.3 `provinces`
`id, region_id (FK), name, code (unique), phu_user_id (FK users NULL)` — the PHU responsible for this province.

### 4.4 `zones`
`id, province_id (FK), name, code`

### 4.5 `areas`
`id, zone_id (FK), name, code`

### 4.6 `parishes`
`id, area_id (FK), name, code, address`

### 4.7 `users`
`id, role_id (FK), first_name, last_name, email (unique), phone, password_hash,
 status ENUM('profile_incomplete','pending_phu','phu_approved','pending_rhu','approved','rejected'),
 profile_complete (bool), email_verified_at, region_id NULL, province_id NULL (denormalized for routing),
 last_login_at, created_by (FK users NULL), deleted_at`

### 4.8 `profiles`
One-to-one with users. All onboarding fields:
`user_id (FK unique), gender, dob, marital_status, whatsapp, home_address,
 region_id, province_id, zone_id, area_id, parish_id, designation,
 occupation, employer, office_address, education,
 year_joined_rccg, year_joined_ushers, ordination_type, ordination_year,
 pastor_in_charge, area_hq_parish, place_of_work,
 passport_url, passport_public_id, signature_url, signature_public_id`
Passport/signature nullable (optional per PRD).

### 4.9 `approval_history`
`id, user_id (FK), stage ENUM('phu','rhu'), action ENUM('approve','reject','correction'),
 actor_user_id (FK), actor_role, comment TEXT, created_at`

### 4.10 `notifications`
`id, user_id (FK), type (varchar), title, body TEXT, channel ENUM('in_app','email','both'),
 read_at NULL, meta JSON, created_at`

### 4.11 `hostels`
`id, name, gender ENUM('Male','Female'), capacity INT, price_kobo INT, description,
 image_url, active (bool)`

### 4.12 `hostel_events` *(supporting table — needed for "Booking Opens/Closes/Active Event")*
`id, name, opens_at, closes_at, active (bool)`
> Added because the PRD's hostel admin needs Event Name + open/close windows. Kept minimal; not counted against the 15 but required.

### 4.13 `hostel_bookings`
`id, booking_number (unique), user_id (FK), hostel_id (FK), event_id (FK),
 region_id, province_id (snapshot), status ENUM('pending_payment','confirmed','cancelled'),
 checked_in_at NULL, created_at`
Unique constraint: one active (non-cancelled) booking per user per event.

### 4.14 `payments`
`id, reference (unique), user_id (FK), purpose ENUM('hostel','id_card'),
 related_id BIGINT (booking_id or id_card_application_id), amount_kobo INT,
 provider ENUM('paystack'), provider_reference, status ENUM('pending','success','failed'),
 paid_at NULL, raw_response JSON, created_at`

### 4.15 `id_card_applications`
`id, application_number (unique), user_id (FK), payment_id (FK NULL),
 status ENUM('submitted','approved','printing','completed'),
 profile_snapshot JSON, passport_url, signature_url,

### 4.16 `audit_logs`
`id, actor_user_id (FK NULL), action (varchar), entity (varchar), entity_id,
 ip, user_agent, meta JSON, created_at`

## 5. Authentication & authorization

- **Register** (self-registerable roles only): First/Last/Email/Phone/Password/Confirm. Creates `users` row with `status='profile_incomplete'`, `role='usher'` by default (self-registration is usher-tier; higher self-registerable designations are captured at onboarding as `designation`, but the *account role* stays usher-level until an admin elevates — this avoids privilege escalation via the public form). → send "registration" notification + email.
- **Login**: verify bcrypt hash → issue access (15 min) + refresh (7 day) JWTs as httpOnly cookies → return user summary. Update `last_login_at`.
- **Refresh**: `/auth/refresh` rotates tokens.
- **Logout**: clears cookies.
- **Forgot/Reset password**: token emailed via Resend, `/auth/forgot`, `/auth/reset`.
- **Change password** (Profile/Settings): current + new + confirm → "password changed" notification + email.
- **Guards**: `requireAuth`, `requireRole('super_admin', ...)`, `requireApproved` (blocks locked modules until `status='approved'`).
- **First-login gate**: if `profile_complete=false` → all `/api` module routes (except profile + auth + hierarchy read) return 403 `PROFILE_INCOMPLETE`; frontend redirects to `/onboarding`.

## 6. Approval routing (the core requirement)

On **profile submission**:
1. User picks `region_id, province_id, zone_id, area_id, parish_id, designation` from real hierarchy dropdowns (cascading, fetched from API).
2. Backend sets `users.region_id` / `province_id` from the submitted profile, `status='pending_phu'`.
3. The responsible PHU = `provinces.phu_user_id` for the chosen province. That PHU's approvals queue is `SELECT users WHERE status='pending_phu' AND province_id = <phu's province>`.

**PHU action:**
- Approve → `status='pending_rhu'`, write `approval_history`, notify user, forward is automatic (RHU derived from region).
- Correction → `status` back to a correction state, comment mandatory, notify + email, user edits, resubmit returns to **PHU only**.
- Reject → `status='rejected'`, reason mandatory.

**RHU queue** = `SELECT users WHERE status='pending_rhu' AND region_id = <rhu's region>` (region derived via `provinces.region_id`).

**RHU action:**
- Approve → `status='approved'` → dashboard unlocked.
- Correction → resubmit returns directly to **RHU**.
- Reject → `status='rejected'`.

No manual forwarding anywhere. PHU/RHU only ever see their own jurisdiction (enforced in SQL, not the client).

## 7. API surface (`/api/v1`)

**Auth**: `POST /auth/register`, `POST /auth/login`, `POST /auth/logout`, `POST /auth/refresh`, `POST /auth/forgot`, `POST /auth/reset`, `POST /auth/change-password`, `GET /auth/me`.

**Hierarchy (public read for cascading selects, admin write)**:
`GET /regions`, `GET /regions/:id/provinces`, `GET /provinces/:id/zones`, `GET /zones/:id/areas`, `GET /areas/:id/parishes`. Admin CRUD under `/admin/hierarchy/*`.

**Profile/Onboarding**: `GET /profile`, `PUT /profile` (save draft), `POST /profile/submit`, `POST /profile/resubmit`, `POST /profile/uploads` (passport/signature → Cloudinary).

**Approvals**: `GET /approvals/queue` (role-scoped), `GET /approvals/:userId`, `POST /approvals/:userId/action` (`approve|reject|correction` + comment).

**Hostel**: `GET /hostels`, `GET /hostel/active-event`, admin `POST/PUT /admin/hostels`, `POST /admin/hostel-events`. `POST /bookings` (create pending), `GET /bookings/me`, `POST /bookings/:id/cancel`, admin `GET /admin/bookings`, `POST /admin/bookings/:id/check-in`.

**Payments (Paystack)**: `POST /payments/init` (purpose + related_id → returns authorization_url/reference), `GET /payments/verify/:reference`, `POST /payments/webhook` (Paystack signature verify → mark success → confirm booking / activate id card → generate QR + receipt → notify + email).

**ID Cards**: `POST /id-cards/apply` (creates app, inits ₦1,000 payment), `GET /id-cards/me`, admin `GET /admin/id-cards`, `POST /admin/id-cards/:id/status`.

**Notifications**: `GET /notifications`, `POST /notifications/:id/read`, `POST /notifications/read-all`.

**Admin (Super Admin)**: `POST /admin/users/phu` (create PHU, link province → auto region), `POST /admin/users/rhu` (create RHU, link region), `GET/POST /admin/hierarchy/*`, `GET /admin/audit-logs`.

Every state-changing endpoint writes an `audit_logs` row.

## 9. Security & config

- Zod-validate every request body/query.
- bcrypt (cost 12) for passwords.
- httpOnly cookies: `Secure`, `SameSite=None` (prod), `SameSite=Lax` (dev). **CSRF**: double-submit token or `SameSite` + custom header check on mutations.
- Strict CORS allowlist (frontend origin) + `credentials: true`.
- `helmet`, rate limiting on auth endpoints, generic auth error messages.
- Paystack webhook signature verification (`x-paystack-signature` HMAC SHA512).
- No secrets in the frontend. `.env` server-side only.

**Backend `.env`**: `PORT, NODE_ENV, DATABASE_URL (or DB_HOST/USER/PASS/NAME), JWT_ACCESS_SECRET, JWT_REFRESH_SECRET, COOKIE_DOMAIN, FRONTEND_ORIGIN, CLOUDINARY_*, PAYSTACK_SECRET_KEY, PAYSTACK_MODE, RESEND_API_KEY, MAIL_FROM`.
**Frontend `.env`**: `VITE_API_URL`.

## 10. Phased execution plan

### Phase 0 — Frontend conversion (TanStack Start → Vite SPA)
1. Add `index.html`, `src/main.tsx`, rewrite `vite.config.ts` to plain `@vitejs/plugin-react` + `@tanstack/router-plugin/vite` + `@tailwindcss/vite` + `vite-tsconfig-paths`.
2. Convert `router.tsx` to browser-history SPA router; remove `shellComponent`/`Scripts`/`HeadContent` SSR bits from `__root.tsx` (use a normal layout; titles via a small head hook or `react-helmet`-style, or keep static).
3. Delete SSR/Nitro/Lovable server files (see §3.1). Remove `@tanstack/react-start`, `nitro`, `@lovable.dev/vite-tanstack-config` from deps.
4. Verify `npm run dev` boots a working SPA, all existing screens render.
**Acceptance:** app runs as SPA, no SSR/server-entry code remains, build passes.

### Phase 1 — Backend foundation
1. Scaffold Express + TS, env validation, mysql2 pool, error/validate/auth middleware, CORS, helmet.
2. Migrations for all tables (§4). Seeds: roles, one Super Admin, sample hierarchy (a few regions/provinces/zones/areas/parishes with assigned PHU/RHU users), sample hostels + one active event.
3. Health check + `GET /auth/me`.
**Acceptance:** DB migrates + seeds, server boots, `/health` green.

### Phase 2 — Auth end-to-end
Backend: register/login/logout/refresh/forgot/reset/change-password/me + cookies + emails.
Frontend: `AuthProvider`, axios client (withCredentials), rewrite `login.tsx` (remove demo accounts, real form), `register.tsx` (correct PRD fields + confirm password), `forgot-password.tsx`, guards.
**Acceptance:** real signup → login → session persists via cookie → logout; wrong password rejected; forgot/reset works.

### Phase 3 — Hierarchy + Onboarding + Profile
Backend: hierarchy read endpoints, profile GET/PUT/submit/resubmit, Cloudinary uploads.
Frontend: `onboarding.tsx` cascading real dropdowns, save draft, submit; `app.profile.tsx` edit + uploads + change password; first-login gate + status banners wired to real status.
**Acceptance:** user completes profile with real hierarchy → status `pending_phu`; uploads land in Cloudinary; incomplete users are gated.

### Phase 4 — Approval routing (PHU → RHU)
Backend: role-scoped queues, action endpoint, `approval_history`, notifications + emails.
Frontend: rewrite `app.approvals.tsx` to hit API, jurisdiction-scoped; status tracker for users.
**Acceptance:** PHU sees only their province's queue; approve forwards to correct RHU by region; correction returns to correct stage; reject requires reason; user is notified at each step.

### Phase 5 — Hostel + Payment + QR receipt
Backend: hostels/events, bookings (one-per-user-per-event), Paystack init/verify/webhook, QR generation, receipt, notifications.
Frontend: rewrite `app.hostel.tsx` — gender-filtered hostels, active event, review → Paystack → success → receipt + QR; admin event/hostel management + bookings table + check-in.
**Acceptance:** male sees male hostels only; pay via Paystack (or mock) → booking confirmed only after webhook/verify → QR receipt emailed; admin sees booking immediately.

### Phase 6 — ID Card
Backend: apply (₦1,000 Paystack), status lifecycle (submitted→approved→printing→completed), digital ID + QR, notifications.
Frontend: rewrite `app.id-cards.tsx` — auto-fill from profile, optional-upload notice + "Update Your Profile" link, pay, status timeline, admin console.
**Acceptance:** apply → pay → status progresses → digital ID with QR; admin updates stages; user notified.

### Phase 7 — Notifications, Audit, Super Admin, scope lockdown
Backend: notifications list/read, audit log middleware on all mutations, PHU/RHU creation, hierarchy CRUD.
Frontend: `app.notifications.tsx` real feed, `app.admin.tsx` real Super Admin console (create PHU/RHU, manage hierarchy, view audit logs). **Strip Phase 2 modules** from `AppSidebar.tsx` + delete/guard their routes. Enforce menu-access rules by status.
**Acceptance:** Phase 2 modules invisible; menu reflects approval status; Super Admin creates PHU (province→region auto-link) and RHU (region); audit logs populated; notifications real.

### Phase 8 — Hardening & handoff
E2E smoke of the full flow (register → onboard → PHU → RHU → dashboard → hostel → id card), rate limits, error states, seed/demo script, READMEs, `.env.example` files, deploy notes.

## 12. Risks / blunt callouts

- **Two servers + cookie auth cross-origin** is the fiddliest part (CORS + SameSite + CSRF). Budget time for it in Phase 2.
- **Paystack** must be webhook-driven; do not mark bookings/ID cards paid from the client. Provide a `mock` mode so development isn't blocked on live keys.
- **The hierarchy seed quality determines whether routing works.** Sample data is fine for MVP, but real go-live needs the real RCCG dataset and correct PHU/RHU assignments.
- Frontend conversion (Phase 0) touches root/router/config; it must land cleanly before any API wiring, or we debug two moving targets at once.
- Money is stored in **kobo (int)** everywhere. Do not introduce float naira.
```
